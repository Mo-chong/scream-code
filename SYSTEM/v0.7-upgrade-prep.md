## 1️⃣ /loop 命令 + shell verifier gate

**变化**：新增 `/loop` 命令，配合 `--verify "command"` 在每轮后跑 shell 检查，exit 0 时自动终止循环。状态栏显示迭代进度。

**当前 fork 状态**：
- CLI 命令路由在 `apps/scream-code/src/cli/` — 纯新增命令，不影响现有命令
- `loop` 相关的代码可能在 `apps/scream-code/src/tui/commands/`

**踩坑点**：
- 🟢 **无冲突** — 纯新增功能，与 fork 现有代码无重叠
- ✅ 可以直接合并

**准备**：
- 合并后验证 `/loop` 在 fork 的构建链下能正常编译
- 确保 `--verify` 的 shell 执行与 Windows/Git Bash 兼容（已有 BashTool 的 Windows 适配可复用）

---

## 2️⃣ npm 更新流 🔴 最高风险

**变化**：
```
旧: git pull + pnpm install + pnpm -r build
新: npm install -g scream-code@latest
```
- 版本检测从 GitHub Releases API → `npm view scream-code version`
- 所有 spawn 调用加 `shell: true`（Windows npm 是 .cmd 文件）
- `detectInstallSource()` 逻辑可能被简化或移除

**当前 fork 位置**：
- `apps/scream-code/src/cli/update/preflight.ts` — `installUpdate()` 做 `git pull + pnpm install + pnpm -r build`
- `apps/scream-code/src/cli/update/source.ts` — `detectInstallSource()` 查 `.git` 目录
- `apps/scream-code/src/cli/update/cdn.ts` — `fetchLatestVersionFromCdn()` 查 GitHub Releases

**核心冲突**：
- 🔴 **更新目标冲突**：`npm install -g scream-code@latest` 安装的是**官方 LIUTod 包**，会覆盖 fork 的所有改动
- 🔴 **fork 的 remote 架构**：origin=LIUTod/scream-code（可 fetch 上游）、mochong=Mo-chong/scream-code（二开 repo）
- 🔴 **版本检测**：fork 的版本号可能不需要与 npm 上的 `scream-code` 同步

**必须做的决策**：
```
方案 A：保留 git 更新流
  - 不合并 v0.7 的 npm 更新代码
  - 保持当前 git pull + pnpm install + pnpm -r build 模式
  - 自己维护 update 相关代码

方案 B：适配 npm 更新流但指向自有包
  - 把 npm package name 改为 fork 自己的包名
  - 需要发布 fork 版到 npm（或私有 registry）
  - 改动较大

方案 C：部分合并
  - 合并 shell: true 等兼容性改进
  - 保留 git 更新模式
  - 不合并 npm 流核心逻辑
```

**准备**：
- 合入前必须在 `preflight.ts` 中识别 v0.7 的更新流代码范围
- 在 `source.ts` 中确认 `detectInstallSource()` 的修改是否影响 fork 的 source 检测
- 测试 Windows 下 `shell: true` 的改动是否影响其他 spawn 调用

## 3️⃣ WolfPack 无限并发

**变化**：WolfPack 不再限制子 Agent 并发数和 item 数，每个请求的 item 都并行 spawn。

**当前 fork 位置**：
- `packages/agent-core/src/tools/builtin/collaboration/wolfpack.ts` — `WolfPackTool`

**踩坑点**：
- 🟡 如果 fork 对 WolfPackTool 有自定义修改（如并发上限、资源控制），需要检查冲突
- 🟡 无限并发可能导致资源耗尽，上游应该有隐式保护（如 AbortSignal）

**准备**：
- 先读 `wolfpack.ts` 确认 fork 有无自定义并发逻辑
- 合并后测试大量 item 场景
- 确认 AbortSignal 链路完整

---

## 4️⃣ 文件工具强化

**变化**：
- Read 工具：ENOENT 时通过后缀名模糊匹配恢复（suffix-match on typos and stale subpaths）
- ReadGroup：容忍缺失路径，继续处理剩余文件
- Bash 拦截：失败后可恢复——agent 会用推荐工具重试而不是死掉

**当前 fork 位置**：
- `packages/agent-core/src/tools/builtin/file/read.ts` — `ReadTool`
- `packages/agent-core/src/tools/builtin/file/read-group.ts` — `ReadGroupTool`
- `packages/agent-core/src/tools/builtin/shell/bash.ts` — `BashTool`

**踩坑点**：
- 🟢 **纯改进** — 不改变接口，只增强健壮性
- Bash 拦截的可恢复性改动涉及 `BashTool.execution()` 返回结果的处理逻辑

**准备**：
- 合并后跑 read / read-group 测试确认
- 确认 Bash 拦截的新 recover 逻辑与 fork 的安全策略不矛盾

**准备**：
- 检查 `MicroCompactionConfig` 的默认值是否被 fork 覆盖
- 合并后验证压缩行为与之前一致

## 7️⃣ 稳定性修复

**变化**：
- stdin EIO dead-terminal 处理
- LSP multi-byte framing + 并发启动修复
- Convergence gate 不再在 Edit 恢复后对冗余验证循环
- fetch URL SSRF guard 增加 DNS rebinding 防护

**当前 fork 位置**：
- `packages/agent-core/src/tools/builtin/lsp-tool.ts` — `LspTool`
- `packages/agent-core/src/tools/providers/local-fetch-url.ts` — SSRF/DNS 检查
- `apps/scream-code/src/cli/` — stdin/EIO 处理
- `packages/agent-core/src/loop/` — convergence gate

**踩坑点**：
- 🟢 **纯 Bug 修复** — 不改变接口，越早合并越安全
- SSRF DNS rebinding 防护在 `local-fetch-url.ts` 的 `assertSafeFetchTarget()` 中
- LSP multi-byte 修复涉及 `lsp/client.ts` 或类似文件

**准备**：
- 这些修复应该**优先合入**，因为它们修复的都是线上 bug
- 合并后跑 LSP 和 fetch URL 测试确认

```

  └── merge → tsc → vitest

第3优先 —— WolfPack 放开并发 (#3) + /loop 命令 (#1)
  ├── 需要先读 wolfpack.ts 确认 fork 自定义
  └── merge → tsc → vitest → 手动测试

第4优先 —— npm 更新流 (#2) ← 需要先做决策
  ├── **必须先选择方案（A/B/C）**

  │   └── merge 时排除 apps/scream-code/src/cli/update/ 目录

  │   └── 需要改 npm package name
  └── 方案C: 部分合并 shell:true 兼容性
      └── merge preflight.ts 的 shell:true 部分，保留 git 流
```
